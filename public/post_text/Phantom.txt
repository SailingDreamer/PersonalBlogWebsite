<b>Problem Statement</b>:
Doppler Phantoms are artificial, specialized devices used in medical imaging to calibrate and test the accuracy of ultrasound machines, particularly measuring blood flow velocity and direction. Unfortunately, the small consumer clique and low demand for doppler phantoms brings about extremely high costs ($2,500-10,000+) makes the barier to entry extremely strong. If the doppler medical device economy is to prosper and prevent monopolization, reducing this high entry cost for each doppler testing phantom would break down this wall 

<b>Mission</b>:
With inspiration from <br>Zurich's DopFlow Open Source Fully-Programmable Flow Phantom for Doppler Ultrasound</br>, I decided to make my own iteration of this project using only 3D printed parts, a medicinal syringe, and a linear actuator from an old 3D printer. 

<b>Requirements</b>:

* Create a connector between the customized linear actuator and the syringe, allowing the piston to be removed easily.
* Ensure the stepper motor has enough torque to displace enough water in little time.
* Mimic various systolic-diastolic heart beats for medical device testing. 
* Create ESP32 serial communication code capable of relaying speeds from the computer to the microcontroller with lower than 10ms delay.
* Ensure syringe system does not surpass front and rear linear actuator boundaries (limit switch override even if code fails)

<b>Programming</b>:

<em>

When analyzing the Zurich Lab’s phantom system and the STM32 microcontroller program’s architecture, we can see that the fundamental communication between master and slave is a transfer of stepper speeds. As opposed to sending an array of wave amplitudes and time intervals from the host computer to the Arduino initially, the system constantly sends hundreds of integer speed values per second. Although this communication may sound computationally taxing, it makes the interface between the existing Zurich software and the Arduino system significantly less troublesome to implement and control. 

For ease of demonstration, I will first introduce the manipulated computer-side program. Within the Zurich Doppler Phantom System's application, the speed_control.py file carries out the imperative control of the GUI and the serial communication system between the slave device and the host computer. Inside of this file, rudimentary changes were made to the send_speed() function. 

<pre>
def send_speed(self, speed, unit: str="m/s"):
    #motor speed is int step per microsec
    STEPS_PER_TURN = 1
    if abs(speed) < 1e-8:
        mot_speed = 0
    else:
        mot_speed = speed*800

    message = str(int(mot_speed+.5))+'\n'
    try:
        self.ser.write(bytes(message, 'utf-8'))
    except: # so that it doesn't crash if disconnected
        return 0
    else:
        return 1
</pre>

For the example's sake, avert your eyes from the first conditional (infinitesimal check), and focus on the else statement. In this simple equation, the normalized speed is scaled into mot_speed by multiplication. The choice of 800 was specific to my stepper motor and its max speed of 800 (if I request a value over this limit the stepper motor will stall). Although this value should be a global constant in the future, it functionally suffices ad hoc. This mot_speed variable then goes through some changes before being converted into a string and serialized. 

Now that we have established the Zurich Lab's method of sending stepper motor speeds over a port and the impromptu changes made to the functionality of the application, we can move to the client-side doppler phantom.

Arduino programs are organized with an initial setup() function called once near the beginning of the program, and a loop() function called continuously. Although I am using an ESP32 microcontroller and not an Arduino, the same underlying program should function for other use cases. Here is the serial listener used to receive the serialized stepper speeds from the aforementioned master-side program:

<pre>
void loop() {
  if ((stepperOne.getCurrentPositionInSteps() < experimentalStepsInFullLoad) || (stepperOne.getCurrentPositionInSteps() > 0)) {
    refillSyringe(stepperOne, backLimitSwitch, frontLimitSwitch, stepperMaxSpeed, stepperMaxAcceleration);
    syringeDir = 1;
  }

  if ((stepperOne.getCurrentPositionInSteps() < experimentalStepsInFullLoad+(trimAdjustRev*stepsInRev))) {
    syringeDir = -1;
  } else if (stepperOne.getCurrentPositionInSteps() > -(trimAdjustRev*stepsInRev)) {
    syringeDir = 1;
  }

  if(Serial.available() > 0) { // <-------------- HERE
    int speed = Serial.readString().toInt();
    if (speed == 0) {
      stopSpeed(stepperOne);
    } else {
      stepperOne.setSpeedInStepsPerSecond(abs(speed));
      runSpeed(stepperOne, (-1*(speed/abs(speed)))*syringeDir);
      Serial.println(syringeDir);
    }

    Serial.println(speed);
  }

  if (!digitalRead(backLimitSwitch) || !digitalRead(frontLimitSwitch)) {
    experimentalStepsInFullLoad = calibrate(stepperOne, backLimitSwitch, frontLimitSwitch);
    refillSyringe(stepperOne, backLimitSwitch, frontLimitSwitch, stepperMaxSpeed, stepperMaxAcceleration);
    syringeDir = 1;
  }
}
</pre>

Please ignore the safety limit switch conditionals inside of the loop function and focus on the if(Serial.available() > 0) conditional. The ESP32 (Arduino) waits for incoming data on the serial port. If a message is sent, the Serial.availible() method returns an integer greater than one, allowing the program to read the collected string and convert it into an integer. This value is then passed to the stepper object's setSpeedInStepsPerSecond() method before having the custom runSpeed() method create a thread to continuously update the stepper motor's driver and ensure constant speed. 

Although this pipeline has been adapted to my specific situation, it works best with minimal changes to the Zurich Lab's complex program, allowing flexible adjustments to waveforms and general ease of use. 
</em>
